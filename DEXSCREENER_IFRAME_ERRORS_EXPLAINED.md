## Common DexScreener Errors

### 1. CORS Policy Errors
```
Access to image at 'https://cdn.dexscreener.com/...' blocked by CORS policy
Access to fetch at 'https://io.dexscreener.com/...' blocked by CORS policy
```

**Why it happens:**
- DexScreener's embedded iframe makes requests to its own servers
- Browser CORS policies block cross-origin access between your app and the iframe
- This is **expected behavior** for third-party iframes

**Impact:** None - the charts still load and work

### 2. Failed to Access iframe Content
```
Could not access iframe content (CORS restriction): SecurityError: 
Failed to read a named property 'document' from 'Window': 
Blocked a frame with origin "http://localhost:5173" from accessing a cross-origin frame
```

**Why it happens:**
- Your code tried to access the iframe's document (for optimization)
- Browser security blocks this access (cross-origin restriction)
- The code gracefully handles this with a try/catch

**Impact:** None - the chart loads normally, just skips some optimizations

### 3. ERR_BLOCKED_BY_CLIENT
```
GET https://io.dexscreener.com/dex/log/exc net::ERR_BLOCKED_BY_CLIENT
GET https://cdn.dexscreener.com/... net::ERR_FAILED
```

**Why it happens:**
- Ad blockers (uBlock Origin, etc.) block tracking/analytics scripts
- Some resources in the DexScreener iframe are flagged as trackers
- Browser extensions prevent these from loading

**Impact:** None - the chart still displays correctly

### 4. 429 Too Many Requests
```
GET https://io.dexscreener.com/hype/reactions/... 429 (Too Many Requests)
```

**Why it happens:**
- DexScreener's rate limiting on their API
- The iframe tries to fetch "hype reactions" data
- Your IP has made too many requests to their API

**Impact:** None - the chart price data still loads

## Why Can't These Be Suppressed?

### They Come From Inside the iframe
- These errors are generated by **DexScreener's JavaScript code** running inside the iframe
- You have **no control** over third-party iframe content
- The iframe is sandboxed for security (as it should be)

### Browser Security Prevents Access
- CORS is a **browser security feature**, not a bug
- It protects users from malicious cross-origin attacks
- You **cannot disable CORS** for security reasons

### Console Shows All Network Activity
- The browser console shows **all network requests**, even from iframes
- There's **no way to filter iframe errors** from your code
- This is by design - it helps developers debug iframe issues

## Developer Tips

### Filter Console Logs
In Chrome DevTools, you can filter out iframe errors:
1. Open Console
2. Click the filter icon
3. Add negative filter: `-dexscreener.com`
4. Add negative filter: `-CORS`
5. Add negative filter: `-Failed to read`

### Focus on Your Code's Errors
Look for errors from **your domain** (localhost:5173, moonfeed.app):
- These are the ones you can actually fix
- DexScreener iframe errors are out of your control

### Production vs Development
In production, you can disable verbose logging:
```javascript
if (import.meta.env.PROD) {
  // Suppress non-critical errors in production
  console.error = () => {};
}
```

But this is **not recommended** - better to just ignore iframe errors.
